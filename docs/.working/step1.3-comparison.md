# Сравнение подходов

## Решение по области выбора
Этот проход по документации не выбирает будущую архитектуру. Он формализует текущую production-codebase как базовую спецификацию для последующей очистки и redesign.

## Сравнение
| Измерение | Текущий базовый срез платформы |
|-----------|---------------------------|
| Frontend | Один Vite-repo с entrypoints `admin` и `site` |
| Backend | FastAPI monolith для admin/public API и бизнес-оркестрации |
| Parser Service | Отдельная FastAPI-среда выполнения для sync/parsing |
| Database | Shared PostgreSQL schema, Alembic migrations только в backend |
| Auth | Cookie-based admin session с refresh |
| Развертывание | Docker Compose локально, Dokploy/Traefik в hosted-среде выполнения |
| **Итого** | **19/30** |

## Рекомендация
**Рекомендуется: текущий базовый срез платформы**

Обоснование:
1. Пользователь явно запросил документацию, которая описывает проект ровно таким, каким он существует сейчас.
2. Замена архитектуры на этапе документации уничтожила бы ценность docs как базового слоя для рефакторинга.
3. Корректный следующий шаг — это точная спецификация текущего состояния границ, контрактов и владения.

## Когда сравнивать альтернативы
Сравнивать альтернативные подходы следует только после того, как базовая документация утверждена и исправлена. Такое сравнение относится к фазе планирования рефакторинга, а не к текущей фазе фиксации архитектуры.
